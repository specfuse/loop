## Adopted-folder lint — FINDING (see Outcome)

```
python3 .specfuse/scripts/lint_plan.py \
  .specfuse/features/INIT-2026-0001-F06-conform-exampleEndpoint-to-validated-spec
```

**FAIL — 5 issues:** WU-01 reported missing all five sections (`Context`,
`Acceptance criteria`, `Do not touch`, `Verification`, `Escalation triggers`).

Root cause is NOT a missing-section issue — #287's body **does** contain all five,
as `## ATX` headings (`## Context`, `## Acceptance criteria`, …). The loop's
`lint_plan.py` section detector matches `^(\**)<section>` (bold `**Context.**` or
plain), and does **not** recognize `## <section>` ATX headings. So a valid
orchestrator issue body, embedded verbatim by adopt, fails the loop linter.

This is a **section-heading-format contract gap** between the two surfaces:
- Orchestrator issue bodies use Markdown ATX headings (`## Context`).
- The loop's WU template + linter use bold (`**Context.**`).

## Outcome

Three of the four mechanisms proved out live against real GitHub:
- **Discovery — PASS.** 13 features enumerated; #287 parsed correctly.
- **Adopt (scaffold) — PASS.** Folder + encoding + body embed worked.
- **Report-back — PASS.** `state:ready → in-progress → done → ready` transitions
  fired exactly and were fully reverted.
- **Adopted-folder lint — FAIL (FINDING).** The assumption from
  GATE-02-REVIEW.md §3 / GATE-03-REVIEW.md Q5 ("issue bodies are well-formed
  five-section WUs the linter accepts") holds for *section presence* but **not for
  section-heading format**: #287 uses `## ATX` headings; the loop linter only
  recognizes `**bold**`/plain. An adopted feature therefore cannot be ground until
  this is reconciled.

**Decision required (human).** The capability's read/adopt/report-back paths all
work; the blocker to a clean end-to-end grind is the heading-format gap. Options:
1. **Broaden `lint_plan.py`** section detection to also accept ATX headings
   (`^(#+\s*|\**)<section>`) — smallest, loop-side; makes adopted folders lint
   clean. (Touches gate-1 code → should be its own WU, e.g. a gate-4 or a
   follow-up `FEAT-2026-0004`.)
2. **Normalize headings in `adopt_feature.py`** when embedding the body
   (`## X` → `**X.**`) — keeps the linter strict but couples adopt to a format.
3. **Fix the orchestrator issue-body template** to emit bold sections — pushes
   the change to the other surface (charter coordination).

Recommendation: option 1 (broaden the linter — ATX is the more standard markdown,
and the loop should accept what real issue bodies use). Carry it as the gate-3
terminal-case follow-up.
