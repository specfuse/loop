Implements the "Feature Prioritization Guidelines" methodology
(`<local-path> Management/Feature
Prioritization Guidelines.gdoc`): five 1-10 criteria (CI, BV, TF, COI,
R), quarterly weights, formula `RS = WCI*CI + WBV*BV + WTF*TF -
WCOI*COI - WR*R`, normalized to a 0-100 Feature Score.

This file owns the **shape**. WU files own their own status; GATE
files own gate status.

## Dependencies

- **Hard depends on FEAT-2026-0010 landing first.** Adds new columns

## Scope OUT

- Cross-repo / orchestrator-level aggregation. Each repo's
  `scoring-criteria.md` and `priorities/<period>.yml` are local;
  orchestrator-level federation is a separate, later feature once the
  orchestrator exists.
- Re-architecting `pick-feature` beyond reading the rank snapshot.
  Today's pick-feature interactive UX stays; only its ranking source
  changes.
- Paying for development work (Cost-of-Implementation in the formula
  stays `COI` derived from Budget bucket only; the broader paid-dev
  scenario the methodology contemplates is out of scope until the
  orchestrator-managed product repos need it).
- A separate UI / dashboard. The rendered ranking lives in
  `.specfuse/roadmap-ranked.md` (git-tracked markdown); no web view.
- Backfilling estimates for FEAT-2026-0001..0010 beyond what the
  ranking-validation step needs (one historical quarter, max).

## Design decisions (locked in this session)

These map to the methodology's open choices. Confirmed by the user
during the FEAT-2026-0010 design discussion.

1. **Cost-of-Implementation (COI) is derived from the Budget bucket**,
   not entered separately. Mapping: `<$5 → 1, $5-25 → 4, $25-100 → 7,
   >$100 → 10`. The formula's `COI` term still pulls its weight; the
   data-entry surface is just one column (Budget).
2. **Weights live in `.specfuse/priorities/YYYY-QN.yml`**, one file
   per quarter, never overwritten. Latest file by name is active.
   History is the audit trail.
3. **Criteria definitions are per-repo, stable.** Each repo authors
   `.specfuse/scoring-criteria.md` defining what each of the five
   criteria MEANS in that project's context. Revisable but rarely
   (carries a `revision:` field + `## Revision log`). For
   specfuse-loop specifically, CI is reinterpreted as "methodology
   user impact" rather than "end customer impact".
4. **Feature Score is normalized to 0-100** via the methodology's
   `a, b` linear remap (`a = 100 / (RS_max - RS_min)`,
   `b = 100 - a*RS_max`).
5. **Re-estimating an `active` feature is allowed**, but
   `roadmap-estimate` requires `--reason "..."` for that case and
   records old→new + reason to the feature's
   `## Estimate revisions` subsection in roadmap.md. Without
   `--reason`, the skill declines. Prevents free score-gaming.
6. **Per-feature scoring data lives in the roadmap detail section
   as a YAML block**, NOT in the table row. The table row stays
   narrow: `ID | Title | Status | Budget | Detail`. Score is
   rendered, never stored.
7. **Audit lives in-detail** (`## Estimate revisions` subsection),
   not in a separate jsonl. Travels with the feature into the
   archive on completion.
8. **Telemetry-grounded estimation:** `roadmap-estimate` reads each
   feature folder's `events.jsonl` to ground CI/COI/R against actual
   past cost / attempts / escalations.
9. **Rendered ranking file: `.specfuse/roadmap-ranked.md`**, git-
   tracked, auto-regenerated by `set-priorities`, `roadmap-estimate`,
   `roadmap-add`, and `roadmap-archive` as their final step. Header
   names the period, weights, and timestamp it was computed under.
10. **Active vs planned features are ranked in separate sections** in
    the snapshot file — active are committed work, not candidates.

## Planned WU outline (prose; flesh out via `/draft-feature` post-0010)

The likely substantive WUs for Gate 1. IDs are tentative — adjust
when drafting against the restructured roadmap shape.

- **T01 — scoring-criteria template + `define-scoring-criteria` skill.**
  Ship `.specfuse/templates/scoring-criteria.template.md` and the
  `/define-scoring-criteria` interactive skill that bootstraps the
  per-repo `.specfuse/scoring-criteria.md`. Includes `--revise` mode
  that appends a `## Revision log` entry on edit.

- **T02 — priorities-file format + `set-priorities` skill.**
  Define `.specfuse/priorities/YYYY-QN.yml` schema (objective +
  five weights). Ship `/set-priorities` that writes a new period
  file (refusing to overwrite an existing one) and reads
  `roadmap-criteria.md` to validate weight keys.

- **T03 — per-feature scoring YAML block + roadmap-detail schema.**
  Extend FEAT-2026-0010's roadmap-add / roadmap-archive skills to
  read/write a `<!-- scoring -->` YAML block in each feature's
  detail section. Block carries CI/BV/TF/R + Budget bucket +
  optional `last_estimated:` timestamp. T03 also adds the
  `## Estimate revisions` subsection convention.

- **T04 — `roadmap-estimate` skill.**
  Interactive skill that fills the scoring YAML block for a feature
  using the criteria rubric + events.jsonl telemetry as grounding.
  For `active` features, requires `--reason "..."` and appends to
  `## Estimate revisions`.

- **T05 — Feature Score computation + `roadmap-rank` skill.**
  Pure computation module (testable in isolation) implementing the
  RS formula + 0-100 normalization. `/roadmap-rank` reads
  `roadmap.md` + the latest `priorities/<period>.yml` + each
  feature's scoring YAML, computes Feature Scores, and emits a
  ranked table. `--snapshot` writes `.specfuse/roadmap-ranked.md`.

- **T06 — Snapshot auto-regeneration hook.**
  Update FEAT-2026-0010's roadmap-add and roadmap-archive (and the
  new T02 / T04 skills) to call `roadmap-rank --snapshot` as their
  final step. Update `pick-feature` to read `roadmap-ranked.md` (or
  fall back to invoking `roadmap-rank` if the snapshot is stale).

- **T07 — Bootstrap specfuse-loop's own scoring data.**
  Write `.specfuse/scoring-criteria.md` (with the methodology-user
  reinterpretation of CI), write `.specfuse/priorities/<current
  quarter>.yml` (initial weights — likely CI heavy while we are in
  adoption-ramp phase), and fill scoring YAML for the current
  planned/active features (0002, 0010, 0011, 0013, 0014). Validate
  the ranking matches Christian's intuitive order for one historical
  quarter.

Plus the four closing WUs (G1-RETRO, G1-LESSONS, G1-DOCS, G1-PLAN).
